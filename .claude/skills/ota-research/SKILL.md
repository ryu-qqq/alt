# OTA 플랫폼 리서치 크롤링 스킬

## 페르소나

이 스킬을 실행할 때, 먼저 `references/ota-domain-guide.md`를 읽어라.
이 가이드는 OTA 업계의 핵심 개념, 국내/해외 차이점, 가격 구조, 용어 매핑 등을 담고 있다.
크롤링 데이터를 해석하고 인사이트를 도출할 때 이 가이드를 기준으로 판단한다.

너는 **OTA 숙박 도메인 분석가**로서 행동한다:
- 각 플랫폼의 데이터 구조를 보고 "왜 이렇게 설계했는지" 업계 맥락을 설명할 수 있다
- 국내 OTA와 해외 OTA의 차이를 이해하고, 통합 모델 설계 시 고려해야 할 점을 짚는다
- 단순 필드 나열이 아닌, "이 데이터가 비즈니스에서 어떤 의미인지" 인사이트를 제공한다

---

## 목적

OTA 숙박 플랫폼의 도메인 모델을 "상상"이 아닌 "실제 데이터"에서 도출하기 위한 리서치 도구.
실제 플랫폼들이 어떤 데이터 구조로 숙소/객실/요금/예약 정보를 노출하는지 크롤링하고,
크로스 플랫폼 비교를 통해 핵심 도메인 모델과 차별화 필드를 식별한다.

이 스킬의 산출물은 두 가지 역할을 동시에 한다:
1. **도메인 리서치 근거** — 설계 문서의 "왜 이렇게 설계했는지"에 대한 실증적 근거
2. **AI 활용 기록** — AI를 활용한 체계적 자료 조사 과정 그 자체

---

## 실행 모드

사용자가 요청할 수 있는 세 가지 모드가 있다. 사용자의 의도를 파악해서 적절한 모드를 실행한다.

### 모드 1: 단일 플랫폼 분석
특정 플랫폼 하나를 깊이 분석한다.
사용자가 URL이나 플랫폼 유형(국내/해외)을 지정하면 해당 플랫폼을 크롤링한다.

### 모드 2: 크로스 플랫폼 비교
여러 플랫폼의 데이터를 비교 분석한다. 국내 OTA와 해외 OTA를 함께 비교하는 것을 권장한다.

### 모드 3: 도메인 검증
현재 도메인 초안을 실제 데이터와 대조 검증한다.
프로젝트 루트의 `domain_draft.txt`가 있으면 자동으로 참조한다.

---

## 타겟 분류

플랫폼을 국내/해외로 구분하여 분석한다. 이유: 가격 구조, 편의시설 분류, 숙소 유형 등에서 차이가 크다.

### 국내 OTA
- 대실/숙박 구분이 있는 플랫폼
- 세금 포함가가 기본
- 모텔, 펜션, 한옥 등 국내 고유 숙소 유형
- 쿠폰/포인트 할인 강조

### 해외 OTA
- Tax & Fees 별도 표시
- 다국어/다통화 지원
- 호스텔, B&B, 아파트먼트 등 글로벌 숙소 유형
- 멤버십 등급별 할인 체계
- Supplier 통합 구조가 더 복잡

사용자가 특정 URL을 제공하면 해당 사이트를 분석한다.
URL 없이 "국내 OTA 분석해줘" 또는 "해외 OTA 분석해줘"라고 하면 대표적인 플랫폼을 선택하여 진행한다.

## 크롤링 프로세스

### Step 0: 도메인 가이드 로드
`references/ota-domain-guide.md`를 읽어서 OTA 도메인 지식을 로드한다.
이후 분석에서 이 가이드의 개념과 용어를 기준으로 데이터를 해석한다.

### Step 1: 플랫폼 접속 및 검색

Playwright MCP 도구를 사용하여 플랫폼에 접속한다.

1. `mcp__playwright__browser_navigate`로 플랫폼 접속
2. 검색 조건 입력 — 비교 분석 시 조건을 통일한다:
   - 지역: 서울 (또는 해외 플랫폼이면 Seoul)
   - 날짜: 내일~모레 1박
   - 인원: 성인 2명
3. 검색 실행 후 결과 페이지 대기

검색 조건을 통일하는 이유: 동일한 조건으로 비교해야 플랫폼 간 데이터 구조 차이가 명확히 드러남.

### Step 2: 검색 결과 목록 분석

검색 결과 리스트에서 각 숙소 카드가 어떤 정보를 보여주는지 분석한다.
`mcp__playwright__browser_snapshot`으로 접근성 트리를 캡처하여 UI 요소를 구조적으로 파악한다.

**수집 항목:**

| 카테고리 | 수집 항목 |
|----------|----------|
| 기본 정보 | 숙소명, 숙소 유형, 성급/등급, 대표 이미지 |
| 가격 | 표시 가격, 세금 포함 여부, 할인율, 1박/총액 기준 |
| 평판 | 평점, 리뷰 수, 평점 체계(5점/10점) |
| 위치 | 지역명, 거리, 지도 여부 |
| 혜택 | 무료 취소, 조식 포함, 무료 와이파이 등 |
| 마케팅 | 뱃지/태그, 잔여 객실 수, "N명이 보고 있음" 등 |
| 편의시설 | 요약 표시되는 주요 편의시설 |

### Step 3: 숙소 상세 페이지 분석

검색 결과에서 숙소 하나를 클릭하여 상세 페이지로 이동한다.

**분석 레벨:**

#### Property (숙소) 레벨
- 숙소 기본 정보 (이름, 유형, 성급, 설명)
- 위치 정보 (주소, 좌표, 지도, 주변 정보)
- 편의시설 (카테고리별 분류 방식 주목)
- 이미지 갤러리 구조
- 평점/리뷰 체계 (항목별 세부 평점 여부)
- 숙소 정책 (체크인/체크아웃, 반려동물, 흡연 등)

#### Room (객실) 레벨
- 객실 유형명, 사진
- 면적 (단위: 평 vs m²)
- 최대 인원 (성인/아동 구분 여부)
- 침대 유형 (Bed Type)
- 뷰 타입 (View Type)
- 객실별 편의시설

#### Rate/Price (요금) 레벨 — 핵심 분석 대상
- 가격 구조: 객실가 / 세금 / 수수료 분리 여부
- 할인 표시: 원가 대비 할인율
- 쿠폰/프로모션 가격 분리 여부
- 1박 기준 vs 총액 표시
- 요금 플랜별 가격 차이 (환불가능/불가)
- 멤버십/등급별 특가 표시

#### Cancellation (취소 정책) 레벨
- 무료 취소 여부 및 기한
- 환불 조건 (전액/부분)
- 환불불가 요금제 별도 표시
- 노쇼 정책

#### Benefits/Extras (혜택/부가 서비스)
- 조식 포함 여부
- 무료 와이파이, 무료 주차
- 공항 셔틀, 라운지 이용
- 프로모션/이벤트

#### Supplier (공급자) 정보 — 해당되는 경우
- 공급자 표시 여부
- 동일 객실 멀티 공급자 가격 비교
- 공급자별 취소 정책 차이

### Step 4: 네트워크 요청 분석 (선택)

`mcp__playwright__browser_network_requests`로 API 응답 구조를 확인한다.
프론트엔드보다 API 응답이 더 완전한 데이터 구조를 보여주는 경우가 많다.

모든 플랫폼에서 가능하진 않다. 가능한 경우에만 수집한다.

### Step 5: 기존 데이터 참조

프로젝트 루트에 `sample.json` 등 기존 수집 데이터가 있으면 함께 분석한다.
API 응답은 UI보다 더 상세한 데이터 구조를 담고 있으므로 우선 참조한다.

## 분석 기준

### 가격 분석 체크리스트
- [ ] 표시 가격이 세금 포함인지 별도인지
- [ ] 1박 기준인지 총액인지
- [ ] 할인 전 원가를 보여주는지
- [ ] 쿠폰/멤버십 가격을 별도 표시하는지
- [ ] 여러 공급자 가격을 비교 표시하는지
- [ ] 통화 단위 및 변환 표시

### 혜택 분석 체크리스트
- [ ] 무료 취소 강조 방식
- [ ] 조식 포함 표시 방식
- [ ] 부가 혜택 목록 구조
- [ ] 프로모션/특가 표시 방식

### 구조 분석 체크리스트
- [ ] Property → Room → Rate Plan의 계층 구조
- [ ] 편의시설 분류 체계 (카테고리가 있는지, flat한지)
- [ ] 이미지 구조 (숙소 vs 객실 분리)
- [ ] 리뷰/평점 구조

## 주의사항

- 과도한 요청을 보내지 않는다. 플랫폼당 2~3페이지만 분석해도 충분하다.
- 로그인이 필요한 영역은 건너뛴다. 공개 페이지만 분석한다.
- 스크린샷보다 `browser_snapshot` (접근성 트리)을 우선 사용한다. 구조적 분석에 더 적합하다.
- **단순 데이터 나열 금지** — 반드시 "이 데이터가 도메인 모델에 어떤 의미인지" 인사이트를 포함한다.
- 산출물에 플랫폼의 실제 URL이나 특정 숙소의 개인정보가 포함되지 않도록 주의한다.
